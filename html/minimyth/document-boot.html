General

MiniMyth2 supports two boot methods:

  • network boot and
  • local boot.

MiniMyth2 supports two root file system types:

  • RAM root file system and
  • NFS root file system.

The primary combination is network boot with a RAM root file system. Network boot with an NFS root file system was added to make development and debugging easier. Local boot was added to support networks that use a DHCP server that cannot be configured to support network boot, and to support installations that desire more security than can be provided when the kernel, root file system and configuration files are downloaded from a TFTP server.

Boot Sequence

Like most Linux distributions, MiniMyth2 must retrieve multiple items when it boots:

  1. the boot loader,
  2. the boot loader configuration,
  3. the kernel,
  4. the initial root file system (if needed),
  5. the final root file system (if needed) and

In some cases, the initial root file system contains everything needed by the system. In these cases, there is no need for a separate final root file system, because the initial root file system contains all the functionality required of a final root file system. For example, when the initial root file system contains all the needed application software, there is not need for a separate final root file system. Essentially, in these cases, the initial root file system is the final root file system.

For MiniMyth2 using a RAM root file system, the RAM root file system is an initial root file system. In addition, there is no final root file system because the RAM root file system contains all the functionality needed to run MiniMyth2.

In some cases, the kernel does not need any extra software in order to retrieve the final root file system. In these cases, there is no need for a separate initial root file system, because the kernel contains all the functionality required of an initial root file system. For example, when all the kernel modules needed to retrieve the final root file system are compiled into the kernel, there is no need for an initial root file system to contain and load the needed kernel modules. Essentially, in these cases, the kernel is the initial root file system.

For MiniMyth2 using an NFS root file system, the NFS root file system is a final root file system. In addition, there is no initial root file system because the kernel contains all the functionality needed to mount an NFS root file system.

The item retrieval is a chained sequence of events:

  1. the computer BIOS retrieves the boot loader,
  2. the boot loader retrieves the boot loader configuration, the kernel and the initial root file system (if needed),
  3. the kernel and the initial root file system retrieve the final root file system (if needed) and
  4. the root file system retrieves the MiniMyth2 configuration.

These items need not be retrieved from the same same location. In fact, these items can be divided into three location groups:

  • the boot loader, the boot loader configuration, the kernel and the initial root file system (if needed),
  • the final root file system (if needed) and
  • the MiniMyth2 configuration.

How and from where these items are retrieved depends on the boot method and the root file system type.

For MiniMyth2 using network boot, the boot loader, the boot loader configuration, the kernel, the initial root file system (if needed) and the MiniMyth2 configuration are retrieved from a server on the network. For MiniMyth2 using local boot, the boot loader, the boot loader configuration, the kernel, the initial root file system (if needed) and the MiniMyth2 configuration are retrieved from the local storage device.

For MiniMyth2 using a RAM root file system, there is no final root file system. For MiniMyth2 using an NFS root file system, the final root file system is mounted over the network.

Boot Line Parameters

You can affect MiniMyth2 with boot line (the APPEND line for PXELINUX and SYSLINUX) configuration. You can find more information in the configuration obtained from the boot line section.

Network Boot

For network boot, you must be able to send boot file name n the DHCP's DHCPOFFER message. Most Windows, MacOS, Linux and BSD based DHCP servers support sending the boot file name. However, most DHCP servers found in off-the-shelf home routers do not support sending the boot file name. Therefore, if you are running a computer based DHCP server, then you will likely be able to configure and send the boot file name. However, if you are running an off-the-shelf home router based DHCP server, then you will likely not be able to configure and send the boot file name.

In addition, for network boot, you must be able to run a TFTP server on your network. There are TFTP servers for Windows, MacOS, Linux and BSD. Therefore, if you have an always-on computer on your network, then you will likely be able to run a TFTP server. However, if you do not have an always on computer on your network, then you will likely not be able to run a TFTP server.

If constraints prevent you from meeting either of these two network boot requirements, then your only option is to use the local boot method.

For network boot, you need a motherboard and network interface adapter that support network boot. While there are other kinds of network boot, the most common is the industry standard PXE (Preboot eXecution Environment) network boot. Most modern motherboards with integrated network interface adapters support PXE network boot. Therefore, this is the network boot method that we will cover.

For PXE network boot, you need a PXE boot loader. While their are other PXE boot loaders, one of the most widely used PXE boot loaders is PXELINUX and one of the most flexible PXE boot loaders is iPXE. Therefore, these are the PXE boot loaders that we will cover.

Network Boot using PXELINUX

For PXELINUX based network boot, the following sequence occurs:

  • The computer BIOS sends a BOOTP request to the DHCP server.
  • The DHCP server tells the computer BIOS the IP address of the TFTP server and the location on the TFTP server of the PXELINUX boot loader, and the BIOS retrieves the PXELINUX boot loader from the TFTP server.
  • The PXELINUX boot loader tells the PXELINUX boot loader the location on the TFTP server of the PXELINUX boot loader configuration file, and the PXELINUX boot loader retrieves the PXELINUX boot loader configuration file from the TFTP server.
  • The PXELINUX boot loader configuration file tells the PXELINUX boot loader the location on the TFTP server of the kernel file and initial root file system file (if needed), and the PXELINUX boot loader retrieves the kernel file and the inital root file system (if needed) from the TFTP server.
  • The PXELINUX boot loader configuration file tells the PXELINUX boot loader, the kernel and the initial root file system (if present) the location of the final root file system file (if needed), and the kernel and initial root file system (if present) mount the final root file system (if needed).
    • For an NFS root file system, the location is the IP address of NFS server and the location on the NFS server of the root file system file.

Givin that most people run their MythTV backend on Linux, the most common scenerio for network boot is a network running the ISC DHCP server, running a Linux TFTP server and using PXELINUX. There are numerous guides on the Internet for setting up this network boot configuration. I think that the one created for network booting an EPIA M motherboard is the best.

However, this guide was not written specifically for MiniMyth2. As a result, the contents of the PXELINUX configuration file and the prefered locations of the kernel and root file system are different. Therefore, for this information, be sure to look at the network boot using PXELINUX with a RAM root file system section or the network boot using PXELINUX with an NFS root file system section, depending on which root file system type you have chosen.

Network Boot using PXELINUX with a RAM Root File System

First, follow the instructions for setting up PXE network boot using PXELINUX found in the network boot using PXELINUX section.

Your PXELINUX configuration file should contain the following, where '{version}' is the version of the MiniMyth2 distribution (e.g. 0.21.0-53): 

DEFAULT minimyth-ram

LABEL minimyth-ram
	KERNEL minimyth-{version}/kernel
	APPEND ro root=/dev/ram0 ramdisk_size=300000 initrd=minimyth-{version}/rootfs

Once you have set up your DHCP and TFTP servers for network boot, you need to download and install the files for booting a RAM root file system by following these steps:

  1. Download the files that are necessary for booting a RAM root file system. These files can be found in the 'boot-image.tar.bz2' file in a MiniMyth2 download page.
  2. Extract the 'boot-image.tar.bz2' file by running the command 'tar -jxf boot-image.tar.bz2. This should create the directory 'ram-minimyth-{version}'.
  3. Change to the user 'root'.
  4. Create a directory named 'minimyth-{version}' in the '{boot}' directory. The '{boot}' directory is the MiniMyth2 boot directory on your TFTP server, which is the directory on the TFTP server that contains the PXELINUX boot loader file ('pxelinux.0').
  5. Copy the 'kernel' file from the 'ram-minimyth-{version}' directory to the '{boot}/minimyth-{version}' directory.
  6. Copy the 'rootfs' file from the 'ram-minimyth-{version}' directory to the '{boot}/minimyth-{version}' directory.

Finally, create an appropriate MiniMyth2 read-only configuration directory in the '{boot}' directory. You can find out how to do this in the MiniMyth2 configuration instructions. If you already have a MiniMyth2 read-only configuration directory be sure that it is up-to-date for the version of MiniMyth you plan to run.

Network Boot using PXELINUX with an NFS Root File System

First, follow the instructions for setting up PXE network boot using PXELINUX found in the network boot using PXELINUX section.

Your PXELINUX configuration file should contain the following, where '{version}' is the version of the MiniMyth2 distribution (e.g. 0.21.0-53), '{nfs-server}' is the IPv4 address of the NFS server and '{nfs-directory}/minimyth-{version}' is the MiniMyth2 root file system shared by the NFS server: 

DEFAULT minimyth-nfs

LABEL minimyth-nfs
	KERNEL minimyth-{version}/kernel
	APPEND ro root=/dev/nfs ip=dhcp nfsroot={nfs-server}:/{nfs-directory}/minimyth-{version}

Once you have set up your DHCP and TFTP servers for network boot, you need to download and install the files for booting an NFS root file system by following these steps:

  1. Download the files that are necessary for booting an NFS root file system. These files can be found in the 'nfs-minimyth-{version}.tar.bz2' file in a MiniMyth2 distribution download directory. The location of the MiniMyth2 distribution download directories can be found in the MiniMyth2 download instructions.
  2. Extract the 'nfs-minimyth-{version}.tar.bz2' file by running the command 'tar -jxf nfs-minimyth-{version}.tar.bz2. This should create the directory 'nfs-minimyth-{version}'.
  3. Change to the user 'root'.
  4. Create a directory named 'minimyth-{version}' in the '{boot}' directory. The '{boot}' directory is the MiniMyth2 boot directory on your TFTP server, which is the directory on the TFTP server that contains the PXELINUX boot loader file ('pxelinux.0').
  5. Copy the 'kernel' file from the 'nfs-minimyth-{version}' directory to the '{boot}/minimyth-{version}' directory.
  6. Extract the 'nfs-minimyth-{version}/rootfs.tar.bz2' file by running the command 'tar -jxf rootfs.tar.bz2' in the 'nfs-minimyth-${version}' directory. This should create the directory nfs-minimyth-{version}/rootfs'.
  7. Move the contents of the 'nfs-minimyth-{version}/themes' directory to the 'nfs-minimyth-{version}/rootfs/rootfs-ro/usr/share/mythtv/themes' directory.
  8. Create a directory named 'minimyth-{version}' in the '{nfs-directory}' directory. The '{nfs-directory}' directory is a directory on your NFS server.
  9. Move the contents of the 'nfs-minimyth-{version}/rootfs' directory to the '{nfs-directory}/minimyth-{version}' directory.
  10. Share the '{nfs-directory}/minimyth-{version}' directory using NFS. This NFS share can (and should) be read only and squash root.

Finally, create an appropriate MiniMyth2 read-only configuration directory in the '{boot}' directory. You can find out how to do this in the MiniMyth2 configuration instructions. If you already have a MiniMyth2 read-only configuration directory be sure that it is up-to-date for the version of MiniMyth you plan to run.

Network Boot using gPXE

For iPXE based network boot, the following sequence occurs:

  • The computer BIOS sends a BOOTP request to the DHCP server.
  • The DHCP server tells the computer BIOS the IP address of the TFTP server and the location on the TFTP server of the iPXE boot loader, and the BIOS retrieves the iPXE boot loader from the TFTP server.
  • The DHCP server tells the iPXE boot loader the URI of the iPXE boot loader configuration file, and the iPXE boot loader retrieves the iPXE boot loader configuration file found at the URI.
  • The iPXE boot loader configuration file tells the iPXE boot loader the URI of the kernel file and the URI of the initial root file system file (if needed), and the iPXE boot loader retrieves the kernel file and the inital root file system (if needed) found at their respecitve URIs.
  • The iPXE boot loader configuration file tells the iPXE boot loader, the kernel and the initial root file system (if present) the location of the final root file system file (if needed), and the kernel and initial root file system (if present) mount the final root file system (if needed).
    • For an NFS root file system, the location is the IP address of NFS server and the location on the NFS server of the root file system file.

Givin that most people run their MythTV backend on Linux, the most common scenerio for network boot is a network running the ISC DHCP server, running a Linux TFTP server and using PXELINUX. There are numerous guides on the Internet for setting up this network boot configuration. Unfortunately, there are few guides on the Internet for network boot on a network running the ISC DHCP server, running a Linux TFTP, running the Apache HTTP server and chainloading iPXE.

Local Boot

MiniMyth2 can boot from a local flash device. It can boot from a USB flash disk connected using USB. It can boot from a CompactFlash card connected using a CF-to-IDE adapter. However, it should be noted that people have had trouble with very long loading times for the MiniMyth2 kernel and MiniMyth RAM root file system when booting from a USB flash disk.

Since MiniMyth2 make no assumptions about the media, it should be able to local boot from other types of local storage devices (e.g. a hard disk) as well. However, we have not tested this and this page does not contain instructions for these configuations.

For local boot, you can use any boot loader that is able to boot Linux from a FAT partition and has inital RAM disk (initrd) support. Boot loaders that meet these requirements include popular boot loaders such as GRUB and LILO. However, in this documentation and the automated installation scripts, we use SYSLINUX, which is a close relative of the PXELINUX boot loader that we use to support network boot using PXELINUX.

Local Boot using SYSLINUX

For SYSLINUX based local boot, the following sequence occurs:

  • The computer BIOS queries the local storage device for the location on the local storage device of the SYSLINUX boot loader.
  • The boot sector on the local storage device tells the computer BIOS the location on the local storage device of the SYSLINUX boot loader file, and the BIOS retrieves the SYSLINUX boot loader from the local storage device.
  • The SYSLINUX boot loader tells the SYSLINUX boot loader the location on the local storage device of the SYSLINUX boot loader configuration file, and the SYSLINUX boot loader retrieves the SYSLINUX boot loader configuration file.
  • The SYSLINUX boot loader contfiguration file tells the SYSLINUX boot loader the location on the local storage device of the kernel file and initial root file system file (if needed), and the SYSLINUX boot loader retrieves the kernel and the initial root file system image (if needed).
  • The SYSLINUX boot loader contfiguration file tells the SYSLINUX boot loader, the kernel and the initial root file system (if present) the location of the final root file system (if needed), and the kernel and initial root file system (if present) mount the final root file system (if needed).

There is no reason that MiniMyth2 cannot local boot the boot loader and kernel while using an NFS root file system. However, we have not tested this and this page does not include instructions for this configuration.

There is no reason that MiniMyth2 cannot local boot with a regular (non RAM) root file system. However, we have not tested and this page does not include instructions for this configuration.

This leaves us with local boot with a RAM root file system.

Given that most people run their MythTV backend on Linux, most people should have a computer running Linux. Therefore, we will give instructions for installing local boot MiniMyth2 using a computer running Linux.

The are two methods for creating a MiniMyth2 local boot device with a RAM root file system: automated and manual. We will cover both.

Local Boot using SYSLINUX with a RAM Root File System (automated install)

The MiniMyth2 distribution contains a shell script that will create a MiniMyth local boot device from a USB connected, flash device. This shell script attempts to detect the flash device and automates the manual steps for creating a MiniMyth2 local boot device with a RAM root file system. To run this shell script, do the following:

  1. If you do not already have one, create a MiniMyth2 read-only configuration directory with your desired files. You can find out how to do this in the MiniMyth2 configuration instructions. If you already have a MiniMyth2 read-only configuration directory be sure that it is up-to-date for the version of MiniMyth you plan to run.
  2. Connect your flash device to the computer running Linux through the USB interface. A USB flash disk can be connected directly. A CompactFlash card can be connected through a USB card reader.
  3. Download the 'mm_local_install' and 'mm_local_helper' shell scripts from the MiniMyth2 distribution download directory's 'scripts' directory. The location of the MiniMyth2 distribution download directories can be found in the MiniMyth2 download instructions.
  4. Run the command 'bash mm_local_install {directory}', where '{directory}' is the directory that contains your MiniMyth2 read-only configuration directory.
  5. Review the information reported by 'mm_local_install' and decide whether or not allow 'mm_local_install' to continue. If the flash device is not correct or 'mm_local_install' was unable to detect the flash device, then you can provide the desired flash device as the second argument to 'mm_local_install'. If the MiniMyth2 distribution download URL is not correct or you want to install a MiniMyth distribution from a different location, then you can provide the URL that points to the desired MiniMyth2 distribution as the third argument of 'mm_local_install'.
  6. Disconnect your flash device from the computer and connect it to your MiniMyth2 frontend.

If you want to see what was installed on the flash device, you can reconnect it and mount it. Essentially, 'mm_local_install' formatted the flash device as FAT, installed the SYSLINUX boot loader and boot loader configuration file, and installed all the files in the 'ram-minimyth-{version}.tar.bz2' file from the MiniMyth2 distribution, including the 'minimyth.md5' file, 'kernel' file, the 'rootfs' file and the 'themes' directory.

Local Boot using SYSLINUX with a RAM Root File System (manual install)

If there is some reason that you cannot perform (or do not wish to perform) an automated install, then you can perform a manual install by following the instructions below:

  1. If you do not already have one, create a MiniMyth2 read-only configuration directory with your desired files. You can find out how to do this in the MiniMyth2 configuration instructions. If you already have a MiniMyth2 read-only configuration directory be sure that it is up-to-date for the version of MiniMyth you plan to run.
  2. Connect your flash device to the computer through the USB interface. A USB flash disk can be connected directly. A CompactFlash card can be connected through a USB card reader.
  3. Download the files that are necessary for booting a RAM root file system. These files can be found in the 'ram-minimyth-{version}.tar.bz2' file in a MiniMyth2 distribution download directory. The location of the MiniMyth2 distribution download directories can be found in the MiniMyth2 download instructions.
  4. Extract the 'ram-minimyth-{version}.tar.bz2' file by running the command 'tar -jxf ram-minimyth-{version}.tar.bz2. This should create the directory 'ram-minimyth-{version}'.
  5. Change to the user 'root'.
  6. Make sure that your flash device is not mounted by running the command 'umount /dev/{device}', where '{device}' is your flash device (e.g. sda).
  7. Format your flash device with the label 'minimyth' and a FAT file system having lowercase and long file name support by running the command 'mkfs.msdos -I -F 32 -n minimyth /dev/{device}'. The FAT file system is used because it is supported by both SYSLINUX and MiniMyth2. The lowercase and long file name support is needed because configuration file names are case sensitive and do not conform to the 8.3 file format. The label 'minimyth' is used because MiniMyth2 identifies any partition labeled 'minimyth' as the MiniMyth boot partition.
  8. Install the SYSLINUX boot loader on your flash device by running the command 'syslinux /dev/{device}'.
  9. Mount your flash device by running the command 'mount /dev/{device} {boot}', where '{boot}' is a mount point of your choosing.
  10. Create the SYSLINUX configuration file '{boot}/syslinux.cfg' containing 
    DEFAULT minimyth-ram

LABEL minimyth-ram
	KERNEL kernel
	APPEND ro root=/dev/ram0 ramdisk_size=300000 initrd=rootfs
  11. Copy the 'kernel' file from the 'ram-minimyth-{version}' directory to the '{boot}' directory.
  12. Copy the 'rootfs' file from the 'ram-minimyth-{version}' directory to the '{boot}' directory.
  13. Copy the 'themes' directory from the 'ram-minimyth-{version}' directory to the '{boot}' directory.
  14. Create the MiniMyth2 distribution file list by running the command 'cd ${boot} ; md5sum `ls -1 kernel rootfs themes/*` > minimyth.md5'. The MiniMyth2 distribution file list will be needed in the future when running the local boot update command 'mm_local_update' to update your MiniMyth2 distribution to the latest version.
  15. Copy your MiniMyth2 read-only configuration directory ('conf') to the '{boot}' directory.
  16. Make sure that everything has been written to your flash device by running the command 'sync'.
  17. Unmount your flash device by running the command 'umount ${boot}'.
  18. Disconnect your flash device from the computer and connect it to your MiniMyth2 frontend.

While we have used the SYSLINUX boot loader, there should be no reason that you cannot use a different boot loader, but we have not tested this. Independent of the boot loader used, you must format the drive/partition as FAT, because that is the only file system recognized by MiniMyth2. Independent of the boot loader used, you the file system must support lowercase and long file names, because configuration files are case sensitive and do not conform to the 8.3 file format.

While we have used a flash device, there should be no reason that you cannot use a different local storage devcie (e.g. a hard disk), but we have not tested this.

While we have used a computer running Linux to perform the manual install, there should be no reason that you cannot use a computer running Windows to perform the manual install, but we have not tested this. The SYSLINUX website has instructions for installing the SYSLINUX boot loader from a computer running Windows. When creating text files such as the SYSLINUX configuration file and the MiniMyth2 configuration file and script, be sure that you use an editor that writes Unix text file format and not DOS text file format.
